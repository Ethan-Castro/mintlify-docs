## 2. MFA failure & recovery

The five MFA outcomes you're most likely to hit at CSI — happy path plus the four common failure modes — each with a phone-mockup illustration and a "What happened / How to fix it" card.

<iframe
  src="/images/interactives/csi_mfa_recovery.html"
  style={{ width: "100%", height: "640px", border: "0", borderRadius: "8px" }}
  title="MFA failure and recovery"
  loading="lazy"
/>

| # | Scenario | What happened | How to fix it |
|---|---|---|---|
| 1 | ✅ Successful login | You set up MFA, verified the factor immediately, and signed in with the correct one-time code. | Nothing to fix. |
| 2 | ✕ "Verify Later" | You tapped "Verify Later" during setup. | Open Account → Security → Multi-Factor and verify the factor now. |
| 3 | ✕ Factor never verified | You added an authenticator app but never finished verification. | Find the pending factor in Multi-Factor Settings, enter a current 6-digit code. |
| 4 | ⚠ Phone clock drift | Your phone's clock drifted, so the one-time code is rejected. | Turn on automatic time on your phone, then retry. |
| 5 | ✕ Locked out | Three incomplete or wrong attempts triggered an account lock. | Open a ticket at hpchelp.csi.cuny.edu — mention the lockout time. |

## 3. Data staging + Slurm pipeline

A 40-second narrated tour of the seven HPCC stages from your laptop to long-term archive. The blue packet hops through, pausing at scratch (which strobes to signal "ephemeral, no backup") and at the SLURM queue (where work piles up before placement).

<Frame caption="The seven-stage HPCC data and execution pipeline. White background, 1920×1080, ~40 seconds.">
  <video controls preload="metadata" src="/images/videos/hpcc_pipeline.mp4" />
</Frame>

| # | Stage | Plain-English narration |
|---|---|---|
| 1 | **Workstation** | You upload code and data from your laptop. |
| 2 | **Transfer / Gateway** | Globus or SFTP moves the files onto the cluster. |
| 3 | **Home** (`/global/u/<id>`) | Files land in your durable home directory — backed up, slow. |
| 4 | **Scratch** (`/scratch/<id>`) | You stage the files into scratch — fast NVMe, no backup. |
| 5 | **SLURM queue** | You submit the job; SLURM queues it until nodes are free. |
| 6 | **Arrow compute** | SLURM places the job on Arrow's CPU or GPU nodes. |
| 7 | **Archive** | Outputs go back to home; long-term results to archive. |

## 4. Hybrid cluster simulation

A scripted 30-second top-down schematic showing how SLURM matches four jobs to CPU, GPU, and high-memory racks. CPU racks are blue squares, GPU racks are purple diamonds, the high-memory node is an amber pill — distinguishable by both color and shape.

<iframe
  src="/images/interactives/hpcc_hybrid_cluster.html"
  style={{ width: "100%", height: "640px", border: "0", borderRadius: "8px" }}
  title="HPCC hybrid cluster scheduling"
  loading="lazy"
/>

| # | Step | Plain-English narration |
|---|---|---|
| 1 | Jobs arrive | Four jobs arrive in the queue. Each says how many CPUs and GPUs it needs. |
| 2 | `job-a` → CPU rack | `job-a` wants 16 CPUs, no GPU. SLURM places it on a CPU rack. |
| 3 | `job-b` → small GPU | `job-b` wants 8 CPUs and 1 GPU. SLURM finds an open GPU rack. |
| 4 | `job-c` waits | `job-c` wants 128 CPUs — bigger than any rack except high-memory. SLURM holds it. |
| 5 | `job-d` → large GPU | `job-d` wants 32 CPUs and 4 GPUs. SLURM puts it on the large GPU rack. |
| 6 | `job-c` → high-mem | `job-c` finally gets the high-memory node. |
